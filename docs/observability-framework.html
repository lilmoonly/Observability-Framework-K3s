Overview

This framework is designed for teams that want a complete, understandable platform baseline without jumping directly into a large managed Kubernetes estate. It provisions K3s, separates worker pools, deploys database backends, centralizes logs, scrapes metrics, imports dashboards, routes alerts, and verifies the result with smoke tests.

Architecture

Control Plane

`k8s-ctrl` runs the K3s API server and is the target for cluster control tasks. The control-plane taint is applied after workers join so K3s bootstrap jobs can schedule during a fresh install.

Worker Pools

Hosts are assigned to named pools using `node_pool`. Roles target pools through node selectors and tolerations, so the framework can grow beyond the reference VM layout without rewriting tasks.

Data Layer

The database role can deploy CloudNativePG PostgreSQL or MongoDB Community through MongoDB Controllers for Kubernetes. MongoDB quorum HA expects three odd-numbered members spread across nodes.

Observability Layer

Prometheus, Grafana, Alertmanager, Fluent Bit, OpenSearch, and the AI detector create a single loop: collect telemetry, visualize state, alert on failure, and flag unusual behavior.

Profiles

The repository keeps one deployable baseline, but operators should think in profiles. A lab profile proves behavior on a laptop. A production profile keeps the same role model and increases capacity, storage durability, secret discipline, and recovery planning.

Lab Profile

  • Use Vagrant and the default 8-node inventory.
  • Use smaller retention windows and storage sizes.
  • Keep one app node and three database nodes.
  • Use `scripts/validate.sh` before deploy and `scripts/smoke-cluster.sh` after deploy.
  • Accept disposable SSH host keys for rebuilt VMs.

Production-Oriented Profile

  • Use real infrastructure inventory instead of Vagrant-only assumptions.
  • Keep secrets in Ansible Vault or an external secret workflow.
  • Use durable storage classes and tested backup targets.
  • Enable strict SSH host key checking for smoke checks.
  • Scale database and app pools before scaling monitoring complexity.
Practical guidance: keep monitoring, logging, and AI on dedicated single nodes in the simple baseline. Scale application and database pools first. Add HA for observability components only after the workload and data layer need it.

Quick Start

1. Start the lab VMs

vagrant up

2. Install Ansible collections

ansible-galaxy collection install -r requirements.yml --force

3. Create and encrypt secrets

cp inventory/group_vars/all/secrets.yml.example inventory/group_vars/all/secrets.yml
ansible-vault encrypt inventory/group_vars/all/secrets.yml

4. Validate locally

./scripts/validate.sh

5. Deploy

ansible-playbook -i inventory/inventory.ini site.yml --ask-vault-pass

6. Smoke test the live cluster

./scripts/smoke-cluster.sh

Configuration

Important files

  • site.yml controls the phase order.
  • inventory/inventory.ini defines hosts and pools.
  • inventory/group_vars/all/main.yml holds non-secret framework settings.
  • inventory/group_vars/all/secrets.yml holds vault-encrypted secrets.
  • cleanup.yml safely removes framework components.

High-impact settings

  • database.type: choose postgresql or mongodb.
  • database.availability_goal: choose dev, replication, or quorum_ha.
  • database.scheduling.spread_policy: choose required, preferred, or none.
  • alerting.*.enabled: enable email, Telegram, Slack, Webex, or PagerDuty receivers.
  • artifact_versions: pin K3s-adjacent tooling and Helm charts.

vagrant destroy -f
vagrant up

./scripts/smoke-cluster.sh
Cleanup safety: `cleanup.yml` never runs unless `cleanup_confirm=true` is provided. Namespace deletion removes PVCs and in-cluster data, so use it only when you intend to reset data. 
# Remove logging stack but keep namespace and PVC data.
ansible-playbook -i inventory/inventory.ini cleanup.yml \
  --ask-vault-pass \
  -e cleanup_confirm=true \
  -e cleanup_scope=logging

# Remove logging stack and delete OpenSearch PVC-backed data.
ansible-playbook -i inventory/inventory.ini cleanup.yml \
  --ask-vault-pass \
  -e cleanup_confirm=true \
  -e cleanup_scope=logging \
  -e cleanup_delete_namespaces=true

Observability

Metrics

kube-prometheus-stack deploys Prometheus, Grafana, Alertmanager, kube-state-metrics, and node-exporter. ServiceMonitors cover Traefik, Fluent Bit, OpenSearch exporter, database metrics, Forgejo, and AI engine where enabled.

Logs

Fluent Bit ships cluster logs to OpenSearch. OpenSearch Dashboards provides the exploration UI, while Prometheus alerts track exporter and logging pipeline health.

AI Signals

The AI anomaly detector reads Prometheus features, writes anomaly events to OpenSearch, and exposes metrics and dashboards for detector health and recent anomaly outcomes.

Runbooks

Pod stuck Pending Scheduling

  1. Run kubectl describe pod -n <namespace> <pod>.
  2. Check taints, tolerations, node selectors, and resource requests.
  3. Check pool capacity with kubectl get nodes -L workload.
  4. If the pod belongs to a framework role, verify the role pool setting in main.yml.

Prometheus target down Metrics

  1. Open Grafana alert details and identify namespace, service, and job.
  2. Run kubectl get servicemonitor -A.
  3. Check endpoint existence with kubectl get endpoints -n <namespace>.
  4. Use SMOKE_ALLOW_UNHEALTHY_TARGETS=true ./scripts/smoke-cluster.sh during known maintenance.

MongoDB not Ready Database

  1. Run kubectl get mongodbcommunity -A.
  2. Check operator logs in the database namespace.
  3. Verify member pods have enough memory and PVCs are bound.
  4. Confirm database.mongodb.members does not exceed usable database nodes when spread is required.

OpenSearch unhealthy Logging

  1. Run kubectl get pods -n opensearch -o wide.
  2. Check memory pressure and PVC usage.
  3. Review OpenSearch logs and exporter metrics.
  4. For lab reset, use cleanup_scope=logging with namespace deletion only if data can be removed.

Ingress not responding Traffic

  1. Check /etc/hosts entries point to 192.168.56.10.
  2. Run kubectl get ingress -A.
  3. Verify Traefik is available in kube-system.
  4. Check backend service endpoints.

Python OpenSSL module error Bootstrap

  1. Rerun Phase 1 so pinned Python packages are repaired.
  2. Confirm the import verification task passes.
  3. Continue the failed phase after Phase 1 finishes.
  4. Avoid mixing apt-managed and pip-managed versions outside the framework pins.

Backup and Restore

The framework creates the platform, but production confidence comes from rehearsed recovery. Treat the commands below as starter runbooks and adapt them to your storage class, retention policy, and backup destination.

PostgreSQL

For production, prefer CloudNativePG-native backup integration with object storage. For a lab export, use `pg_dump` through a temporary client pod or a controlled exec into the cluster. 

kubectl get cluster.postgresql.cnpg.io -n database
kubectl get pods -n database -l cnpg.io/cluster=app-db-cluster

MongoDB

Use `mongodump` against the replica set service with the app or admin credentials. Restore with `mongorestore` after validating the target replica set is healthy. 

kubectl get mongodbcommunity -n database
kubectl get pods -n database -l app=app-mongo-rs-svc

OpenSearch

Production OpenSearch should use snapshot repositories. The lab can be reset through namespace and PVC deletion, but that is not a backup strategy. 

kubectl get pvc -n opensearch
kubectl logs -n opensearch -l app.kubernetes.io/name=opensearch

Grafana

Framework dashboards are vendored in git, so the main state to preserve is custom UI changes and credentials. Prefer managing dashboards as code whenever possible. 

kubectl get configmap -A -l grafana_dashboard=1
kubectl get secret -n monitoring

Release Checklist

Before merging changes

  • Run ./scripts/validate.sh.
  • Review changed playbooks for idempotency.
  • Confirm no real secrets are committed.
  • Check the GitHub Actions validation workflow.

Before calling a release stable

  • Run a scratch redeploy from destroyed VMs.
  • Run ./scripts/smoke-cluster.sh.
  • Open Grafana and check dashboards load.
  • Confirm alerts are normal or understood.
Current recommendation: freeze playbook changes once a scratch redeploy and smoke validation pass. Start new work in a dedicated milestone so production confidence does not get blurred by unrelated improvements.
